Overview

What the app does, in three sentences.

The user opens the app, picks a lesson from the sidebar, and steps through slides with l (forward) and h (back) — exactly the motions Vim uses for cursor movement. Some slides are text; others are drills — a small editor that the user has to transform from a starting state to an expected state, several times in a row. Results are saved to disk so lesson completion persists across launches.

Essentials build This branch (essentials) contains only the core learning loop — roughly 1,000 lines across 20 files. The master branch adds a mode indicator (observes kindaVim via distributed notifications), a progress inspector, achievement rings, telemetry-grade drill recording, file-based logging, and a six-chapter curriculum. Everything here exists in master too; nothing was added for the essentials branch.

Architecture

Three layers, one direction of dependency.

Each layer depends only on the layers below it. No upward references.

Why this shape

A common SwiftUI pitfall is letting business logic leak into views — onAppear handlers that mutate state, timers embedded in ViewBuilders, validation logic scattered across observers. This app keeps all of that in the Engine layer. ExerciseEngine is the single source of truth for one drill; views just show what it already computed.

Persistence sits in Services (ProgressStore), which is injected into views that need it. Because state is @Observable, any view that reads a property automatically re-renders when it changes — no manual bindings, no ObservableObject boilerplate.

Data flow

What happens when you press a key during a drill.

One keystroke → AppKit → Coordinator → Engine → observable mutation → SwiftUI re-render.

The dashed arrow from Engine → View is the Observation framework's magic. DrillStepView never subscribes explicitly; the Swift compiler tracks which observable properties the view's body reads and re-runs body when any of them change. The persistence write (bottom arrow) only happens once, when the final rep of the drill completes.

Key concepts

The whole curriculum is a compile-time constant

Open Curriculum/Chapter1_SurvivalKit.swift. It's a Swift file full of literal Exercise(id: ..., initialText: "...") calls. There is no server, no JSON, no CMS. Changing a lesson means editing the file and recompiling. This keeps the tutor simple, offline-only, and lets you refactor content with Swift's type checker.

An Exercise is "initial state → expected state"

Every drill is defined by four values: the initial text, the initial cursor position, the expected text, and the expected cursor position. Success is simply currentText == expectedText && currentCursor == expectedCursor. That's the whole validation rule (ExerciseEngine.validate(currentText:cursorPosition:)).

Variations let one exercise reuse the same task across reps but change the concrete strings, so the user can't pass a drill by muscle memory on a single input.

ContentBlock is a small DSL

Rather than templating lesson explanations as HTML or markdown, the app describes them as an array of enum cases: .heading, .text, .tip, .keyCommand, .codeExample. ContentStepView switches over the cases and renders each. Adding a new kind of explanation means adding an enum case and a new case in the switch — the compiler tells you both places to update.

LessonStep is the presentation of a lesson

Lessons in the data model are unordered bags of content + exercises. LessonStep.steps(from:chapterTitle:) flattens a lesson into an ordered sequence of slides:

  1. A .title step (chapter + lesson name)
  2. One .content step per heading-delimited group of blocks
  3. One .drill step per exercise

This decouples "what the lesson says" from "how it's presented". The step-based canvas only knows about steps; it never reaches into a raw Lesson.

Progress is keyed by exercise ID

UserProgress.completedExercises is [String: ExerciseResult]. Only the best result is retained — fewer keystrokes wins, ties broken by faster time. Because IDs are stable ("ch1.l1.e1"), renaming a lesson doesn't lose the user's progress; renaming an exercise ID would.

Reading the code

A guided tour. Follow these steps with the code open and you'll understand the whole app in 15 minutes.

  1. Start at the entry point Sources/KindaVimTutor/App/KindaVimTutorApp.swift is the @main struct. Everything flows from here. Note that it only hosts a NavigationSplitView — no business logic, no timers, no listeners.
  2. Look at root state AppState is @Observable, @MainActor. It owns the curriculum and the selected lesson. The app binds selectedLessonId into the sidebar so List(selection:) can mutate it directly.
  3. Follow the sidebar selection SidebarView receives a @Binding var selectedLessonId. When the user clicks a row, List writes to it, which mutates AppState, which causes the detail pane to re-render with the new lesson.
  4. Enter the detail pane StepCanvasView holds its own @State private var controller = LessonStepController(). On onAppear (and on lesson change), it loads the lesson into the controller, which runs LessonStep.steps(from:) to produce the slide sequence.
  5. Understand step transitions The canvas renders controller.currentStep inside a switch. Each case picks a view. The .transition(...) modifier combined with .id(step.id) gives SwiftUI the hint to slide-animate between steps.
  6. Hit a drill DrillStepView creates an ExerciseEngine and calls engine.start(exercise) on appear. It hosts ExerciseEditorView (an NSViewRepresentable wrapping NSTextView) and wires the text/selection callbacks into the engine.
  7. Trace a keystroke Open ExerciseEngine.textDidChange(currentText:cursorPosition:). It increments a counter, then calls validate. validate compares against currentVariation.expectedText; on a match, completeRep transitions state and starts the next variation.
  8. See the completion side effect Back in DrillStepView, onChange(of: engine.isDrillComplete) builds an ExerciseResult and calls progressStore.recordCompletion(...). ProgressStore writes JSON to ~/Library/Application Support/kindaVimTutor/progress.json.
  9. Confirm the sidebar updates Next time SidebarView's body runs — which happens when anything it observes changes — progressStore.isLessonCompleted(lesson) returns true and LessonRowView renders a checkmark. You didn't wire this; Observation did.

Beginner macOS developer notes

If you're coming to this codebase new to Swift/SwiftUI, here are the APIs you'll see on every page.

@Observable and the Observation framework

The @Observable macro (Swift 5.9+) lets you define a plain reference type whose properties SwiftUI automatically tracks. Replacing the old ObservableObject + @Published protocol. Views don't declare what they want to observe — the compiler inspects the body and subscribes only to the properties that are actually read. This is why AppState, ExerciseEngine, and ProgressStore don't need any @Published markers.

@Observable
@MainActor
final class ExerciseEngine {
    private(set) var state: State = .idle
    // Reading `state` in any View.body subscribes automatically.
}

@State, @Binding, and ownership

In a View, @State means this view owns this value for its lifetime. @Binding means something above me owns it; I get read/write access. In StepCanvasView, @State private var controller = LessonStepController() — the canvas owns the controller. In SidebarView, @Binding var selectedLessonId: String? — AppState owns the value; the sidebar mutates it through the binding.

@MainActor

Swift's actor system protects against data races. @MainActor means "this runs on the main thread" — which is required for any UIKit/AppKit/SwiftUI work. Every state type in this app is @MainActor; views are implicitly main-actor too. You'll see MainActor.assumeIsolated { ... } used to bridge from AppKit callbacks that Swift can't prove are main-actor.

NavigationSplitView

The two-column layout. First closure is the sidebar; detail: is the right pane. List(selection:) inside the sidebar binds automatically to the selected lesson ID — this is why just changing a @State String? updates the detail view.

NSViewRepresentable — bridging AppKit

SwiftUI doesn't ship a text view as capable as NSTextView (rich editing, accessibility identifiers, custom insertion point color). ExerciseEditorView wraps one. Three methods to know:

  • makeNSView(context:) — build once
  • updateNSView(_:context:) — called when SwiftUI inputs change; reconcile
  • makeCoordinator() — create a helper class that can be the NSTextViewDelegate and hold closures

The Coordinator pattern is how you carry state that AppKit needs (delegates, selectors, cached closures) across the value-type SwiftUI boundary.

DistributedNotificationCenter (not in essentials; in master)

macOS apps can post system-wide notifications that any other process can observe. kindaVim posts kindaVimDidEnterNormalMode etc. when the user changes modes. The stripped build doesn't use this, but if you look at the master branch's ModeMonitor, that's what it's listening to.

FileManager and Application Support

User data that isn't a "document" lives in ~/Library/Application Support/<bundle-id>/. Look up the URL with FileManager.default.urls(for: .applicationSupportDirectory, in: .userDomainMask). The directory may not exist on first launch; always createDirectory(at:withIntermediateDirectories: true) before writing. See ProgressStore.save().

Swift Package Manager (no Xcode project)

There is no .xcodeproj in this repo. Instead, Package.swift declares a single executable target. You can open the Package.swift file directly in Xcode — it'll resolve dependencies, show previews, and build just like a project. From the terminal, swift build and swift test work. The trade-off: no Asset Catalog, no Storyboards, and a slightly more manual setup for things like app bundles (see Scripts/).

SwiftUI previews (#Preview)

Every view file in this project ends with a #Preview("...") { ... } block. With the package open in Xcode, the canvas renders these live. The Views/PreviewSamples.swift file holds shared fixture data so previews stay short. Use previews aggressively — they're the fastest feedback loop in SwiftUI.

Tip When reading a new SwiftUI codebase, go file-by-file in the order: App → State → Models → Engines → Views. That tracks data flow rather than folder names, and the dependency arrows only ever point downward.
