## What "the help system" is

Neovim help is not a single file. It is a tagged documentation system made of many plaintext files under `runtime/doc`, plus generated tag indexes.

Core pieces:

- `help.txt`: main top-level table of contents for builtin docs
- `helphelp.txt`: how help lookup, navigation, tags, translated help, and help authoring work
- `index.txt`: index of commands/topics
- `usr_toc.txt` and `usr_*.txt`: the task-oriented user manual
- topic files such as `motion.txt`, `change.txt`, `options.txt`, `lua.txt`, `lsp.txt`, `api.txt`
- generated `tags` files: lookup tables built from `*tag*` markers inside help files

## How help lookup works

The main user-facing entrypoint is `:help`:

- `:help` opens the help window
- `:help {subject}` jumps to the best-matching help tag
- `:help!` tries to infer a useful tag from the word under cursor
- `CTRL-]` follows a tag under cursor
- `CTRL-O` or `CTRL-T` jumps back
- `:helpgrep {pattern}` searches help text content
- `:helptags {dir}` generates the `tags` index for a help directory

Help subjects are context-sensitive. `helphelp.txt` documents common prefixes:

- plain `:help x` for a Normal-mode command
- `:help v_u` for a Visual-mode command
- `:help i_<Esc>` for Insert-mode behavior
- `:help :quit` for Ex commands
- `:help 'textwidth'` for options
- `:help /[` for regexp syntax

Matching is tag-based, not full-text by default. The `tags` file is generated by scanning help files for `*tag-name*` markers.

## How the help files are written

The source files are regular text with lightweight markup conventions:

- `*tag*` defines a jump target
- `|tag|` links to another tag
- `'option'` refers to an option
- backticks mark inline code-ish literals
- section separators are lines of `=`
- headings often end with `~`
- modelines set `ft=help`

This is why the corpus stays readable in git while also supporting structured in-editor navigation.

## What the corpus contains

The current snapshot includes:

- reference docs for editing, motions, options, mappings, windows, tabs, search, quickfix, diff, terminal, syntax, spell, folds
- user-manual chapters in `usr_*.txt`
- Neovim-specific docs for Lua, API, UI events, LSP, Treesitter, diagnostics, jobs/channels, remote plugins, providers
- plugin and filetype help files
- news and deprecation docs
- developer docs under `dev*.txt`

One practical implication: if you want "the contents of the help file", the closest real answer is "the contents of the whole `runtime/doc` corpus", with `help.txt` as the top-level index rather than the whole system by itself.
